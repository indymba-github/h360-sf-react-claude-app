# v2.3.0 FINS Relationship Workspace Design

## Goal

Make the account detail page feel like a practical FINS relationship workspace: polished enough to support a demo conversation, but dense enough to help inspect a real customer relationship.

The page should answer four questions quickly:

- What is the relationship worth?
- Where is growth or pipeline momentum?
- Where is service friction?
- What should I ask or inspect next?

## Current State

The account detail page already loads the right data:

- Account profile and hero stats
- Open and closed opportunities
- Contacts
- Cases
- News alerts
- Financial accounts
- Agentforce briefing chips
- AI chat with account context
- Rendered AI tools, including account risk briefing

The main gap is narrative cohesion. The data is present, but the page reads as separate sections rather than a relationship view.

## Recommended Approach

Add a deterministic **Relationship Snapshot** near the top of the account detail page, below the page heading and above the existing Agentforce briefing chips.

This is the recommended approach because it uses data already fetched by the page, avoids adding new API dependencies, and improves the first impression without making AI response quality the single point of polish.

## Alternatives Considered

### AI-First Briefing

Make generated AI or Agentforce summaries the dominant experience.

This is attractive for demos, but it depends on API latency, prompt quality, and provider availability. It is better as a follow-up action than the primary account framing.

### Dense Data Cleanup

Polish each existing section independently.

This would help, but it would not solve the bigger issue: the user still has to mentally assemble the relationship story from separate blocks.

## Scope

The v2.3.0 slice should add:

- A `RelationshipSnapshot` component on `/accounts/[id]`
- Deterministic FINS-oriented signal cards
- A small next-action area with account-scoped AI prompt chips
- Empty and partial-data states that still feel intentional
- Focused tests for the summary helpers and/or component behavior

## Relationship Signals

The first version should compute signals from data already available on the account detail page.

### Relationship

Use financial account records to show:

- Total financial accounts
- Product mix count
- Known balance or value summary when available

If financial data is missing, show a calm empty state instead of implying failure.

### Growth

Use opportunities to show:

- Open pipeline total
- Open opportunity count
- Next close date
- Highest-probability open opportunity when available

### Service

Use cases to show:

- Open case count
- High-priority open case count
- Most recent case date when available

### Coverage

Use contacts to show:

- Contact count
- Email coverage
- Phone coverage
- Simple gap message when contact data is thin

## AI Follow-Up

Add two to four prompt chips that dispatch the existing `chat:ask` event.

Initial prompts:

- Explain this relationship
- Where is the growth opportunity?
- What service risks should I watch?
- What should I do before the next meeting?

The prompts should be scoped to the current account and should not introduce a new chat pathway.

## Placement

Place the Relationship Snapshot:

1. After the page heading and Salesforce record link
2. Before Agentforce briefing chips
3. Before News, Pipeline, Contacts, Financial Accounts, and Cases

This keeps the page anchored in Salesforce data while giving the user a useful orientation layer before the detailed records.

## Component Boundaries

Add a small data-shaping helper so the component receives a clean snapshot model rather than raw Salesforce records.

Suggested boundaries:

- `dashboard/lib/relationship-snapshot.ts`
  - Computes deterministic snapshot signals from account, opportunities, contacts, cases, and financial accounts.
- `dashboard/components/account/RelationshipSnapshot.tsx`
  - Renders the snapshot and prompt chips.
- `dashboard/app/accounts/[id]/page.tsx`
  - Calls the helper and inserts the component.

## Error Handling

The snapshot should tolerate partial data because the existing page already uses `Promise.allSettled`.

If one data source fails:

- Show the available signals
- Mark the missing signal as unavailable
- Do not block the whole account page
- Do not claim data is zero when it is actually unavailable

## Testing

Add focused tests for deterministic signal calculation.

Coverage should include:

- Full data available
- No financial accounts
- Open and closed opportunity mix
- Open and high-priority cases
- Missing contact email or phone coverage
- Partial unavailable data source

If practical, add or extend a smoke test that confirms the account page renders the Relationship Snapshot without client-side errors.

## Self-Review

- No hardcoded Salesforce IDs, credentials, or environment-specific values.
- Scope is limited to account detail workspace polish.
- The design does not require new provider credentials or new external services.
- The AI chips reuse the existing chat event path.
- The release process remains gated until implementation and validation are complete.
